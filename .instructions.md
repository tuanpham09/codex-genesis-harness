# Genesis Harness - Workflow Instructions

**Purpose**: Minimal, focused hooks to optimize token usage while maintaining system integrity and developer control.

---

## PreToolUse Hooks

**Trigger**: Before ANY file creation, modification, or complex command

### 1. Load Context (300ms - one-time per session)
```
If .codebase/CURRENT_STATE.md exists:
  → Load into context (max 50 lines)
  → Skip if already loaded this session
  
Purpose: Know current state before making changes
Impact: Prevents context duplication, saves ~15% tokens
```

### 2. Validate File Paths
```
For any file operation:
  IF file path contains /contracts/ or /fixtures/
    → Verify corresponding .md contract exists
    → Suggest contract creation if missing
  
  IF file path contains /.planning/ or /.codebase/
    → Verify YAML/Markdown syntax valid
    → Suggest format correction if needed
    
Purpose: Catch issues early, prevent rework
Impact: ~5% token savings on retry prevention
```

### 3. Check Memory Files
```
Before task execution:
  IF .codebase/MODULE_INDEX.md exists
    → Skip loading full file listing (too expensive)
    → Reference path pattern instead
  
  IF .planning/SPEC_CHANGELOG.md exists
    → Don't re-read unless explicitly needed
    → Assume current unless user says "changed"
    
Purpose: Trust cached state, avoid redundant reads
Impact: ~20% token savings on memory access
```

### 4. Auto-Trigger Research-First (NEW - Critical)
```
For any important task:
  IF task_type in [init, new-feature, fix-bug, spec-change, architecture, design, plan, upgrade]:
    
    THEN Activate genesis-research-first skill:
      1. Research Phase (10-15 min):
         - Read local documentation (.codebase/*, README, contracts)
         - Search codebase for similar patterns
         - Research external sources (GitHub, official docs)
      
      2. Compile Research Note:
         - Question: What are we deciding?
         - Local Evidence: From codebase
         - External Evidence: From research
         - Recommendation: Based on evidence
         - Risks: Identified constraints
      
      3. Generate Initial Plan:
         - Pre-populate with research findings
         - Pre-populate with existing patterns
         - Pre-populate with risks/constraints
      
      4. Pass to User:
         - Present research note + plan for review
         - Wait for user approval before proceeding

Purpose: No guesswork. Every decision grounded in evidence.
Impact: Eliminates "wrong assumptions", 100% best-practice alignment
Rules: 
  - NEVER skip research for "simple" tasks
  - Research is automatic, not optional
  - Research completes BEFORE planning
  - Planning is auto-generated FROM research output
```

## PostToolUse Hooks

**Trigger**: After file creation/modification succeeds

### 1. Auto-Sync Memory (Priority 1)
```
IF files modified in .planning/ OR .codebase/:
  → Check if SPEC_CHANGELOG.md needs update
  → Check if MODULE_INDEX.md needs update
  
Action: Create entry only, don't rewrite entire file
Format:
  - Date: YYYY-MM-DD HH:MM:SS
  - File: relative path
  - Change: 1-line summary
  
Example entry:
  - 2026-05-30 14:30: .codex/skills/spec-impact-engine/SKILL.md - Added new spec detection workflow

Purpose: Keep memory auto-synchronized
Impact: 100% memory consistency, saves ~10 min per session
```

### 2. Validate Syntax (Priority 2)
```
IF markdown or YAML created/modified:
  → Quick syntax check (line count, bracket matching)
  → Report any obvious issues
  → Don't run full linters (too expensive)
  
Purpose: Catch syntax errors before commit
Impact: Prevents CI/CD failures, saves rework
```

### 3. Update Affected References (Priority 3)
```
IF API_DOCS.md OR ARCHITECTURE.md OR DESIGN.md modified:
  → Note in SPEC_CHANGELOG.md what changed
  → Suggest which tests to review
  → DON'T auto-update tests (developer chooses)
  
Purpose: Awareness, not automation
Impact: Developer stays informed
```

### 4. Auto-Trigger Debug Verification After Bug Fix (Priority 1)
```
IF /fix-bug completes successfully:
  → Activate genesis-debug-guide skill
  
  DEBUG VERIFICATION WORKFLOW:
  1. Fix Verification (5 min):
     - Run full test suite
     - Verify: All tests pass
     - Verify: No debug logs in changed files
     - Verify: No dead code introduced
  
  2. Coverage Check (3 min):
     - Confirm: Test exists for this bug
     - Confirm: Test fails without fix
     - Confirm: Test passes with fix
     - Confirm: Related tests still pass
  
  3. Regression Prevention (5 min):
     - Search: Similar patterns in codebase
     - Add: Tests for edge cases found
     - Document: In .codebase/RECOVERY_POINTS.md
     - Scan: Other instances of same issue
  
  4. Completion (2 min):
     - Create: Debug investigation log (templates/debug-investigation-log.md)
     - Update: .codebase/CURRENT_STATE.md with fix status
     - Mark: Bug fix verified and ready for commit

Purpose: No manual verification needed - automatic quality gate after every bug fix
Rules:
  - Genesis-debug-guide auto-triggers AFTER /fix-bug completes
  - Not optional - part of /fix-bug completion criteria
  - Ensures: Tests pass, no regressions, fix quality verified
  - Output: Investigation log + verification evidence
Impact: 100% fix verification, zero manual steps, regression prevention built-in
```

### 5. Auto-Trigger Spec Propagation After Spec Change (Priority 1)
```
IF /spec-change completes successfully:
  → Activate genesis-spec-propagation skill
  
  SPEC PROPAGATION WORKFLOW:
  1. Change Detection (5 min):
     - Analyze: Old spec vs new spec
     - Identify: Breaking vs feature vs non-impact
     - Map: Affected phases (2, 3, 4, 5)
     - Record: Entry in SPEC_CHANGELOG.md
  
  2. Downstream Impact Analysis (10 min):
     - Build: Dependency graph (Phase 1 → 2 → 3 → 4 → 5)
     - Trace: Propagation paths
     - Identify: Update type per phase
     - Check: For conflicts/dependencies
  
  3. Auto-Update Phases (15 min):
     - Phase 2: Update test mocks + assertions
     - Phase 3: Update API contract + handler
     - Phase 4: Update client SDK types
     - Phase 5: Update E2E scenarios + tests
     - Generate: Migration guide (if breaking)
  
  4. Validation (5 min):
     - Syntax: All files valid JSON/JS/TS
     - Cross-phase: Types aligned + consistent
     - Tests: Run affected test suites
     - Ready: For code review
  
  5. Completion (2 min):
     - Create: SPEC_CHANGE_PROPAGATION.md log
     - Update: .codebase/CURRENT_STATE.md
     - Gate: BREAKING changes require manual approval
     - Mark: Phases aligned and ready for commit

Purpose: Auto-sync all phases when spec changes - prevent cascading rework
Rules:
  - Genesis-spec-propagation auto-triggers AFTER /spec-change completes
  - Not optional - part of /spec-change completion criteria
  - BREAKING changes: Require manual review gate before commit
  - Ensures: All phases aligned, no stale contracts, migration guides complete
  - Output: Updated phase files + propagation log + migration guide
Impact: 45-60 min time savings per spec change, prevents phase misalignment, zero manual coordination
```

### 6. Auto-Trigger Docs Automation After Tests Pass (Priority 2)
```
IF tests pass successfully (Phase 2 or Phase 5):
  → Activate genesis-docs-automation skill
  
  DOCS AUTOMATION WORKFLOW:
  1. Change Detection (5 min):
     - Find: All files changed since last stable state
     - Categorize: By phase (1, 2, 3, 4, 5)
     - Classify: Change type (FEATURE, BUG_FIX, BREAKING, INTERNAL)
     - Map: To documentation update types
  
  2. Doc Type Analysis (10 min):
     - API Contract changed? → Generate API_REFERENCE.md section
     - Backend changed? → Generate IMPLEMENTATION.md section
     - SDK changed? → Generate SDK_REFERENCE.md section
     - Tests changed? → Generate test coverage notes
     - All changes? → Generate SPEC_CHANGELOG.md entry + IMPLEMENTATION_HANDOFF.md
  
  3. Auto-Update Documents (15-20 min):
     - API_REFERENCE.md: Add/update endpoint docs
     - IMPLEMENTATION.md: Add/update implementation details
     - ARCHITECTURE.md: Update if data flow changed
     - SPEC_CHANGELOG.md: Add entry with migration guides (if breaking)
     - IMPLEMENTATION_HANDOFF.md: Update phase status and next steps
     - Generate: Migration guides (if breaking changes)
  
  4. Validation (5 min):
     - Syntax: All markdown files valid
     - References: All internal links exist + accurate
     - Cross-phase: Types aligned, error codes consistent
     - Completeness: All endpoints/methods documented
     - Breaking changes: Flag for manual review gate
  
  5. Completion (2 min):
     - Create: DOCS_UPDATE_LOG.md entry
     - Update: .codebase/CURRENT_STATE.md with docs status
     - Gate: BREAKING changes require manual approval
     - Mark: Docs ready for commit or flagged for review

Purpose: Keep documentation in sync with code - prevent documentation drift
Rules:
  - Genesis-docs-automation auto-triggers AFTER tests pass
  - Not optional - part of test phase completion criteria
  - BREAKING changes: Require manual review gate before commit
  - Ensures: Docs always current, no stale references, migration guides generated
  - Output: Updated doc files + changelog entry + update log
Impact: 30-60 min time savings per phase, prevents documentation drift, zero manual doc sync
```

## Stop Hooks

**Trigger**: When session ends (user closes chat or says "done")

### 1. Session Verification (Always run)
```
Verify state:
  ✓ All modified files have .md extensions (if applicable)
  ✓ No syntax errors in created files
  ✓ SPEC_CHANGELOG.md has today's entries
  ✓ No uncommitted changes reported
  
Report: "✓ Session state verified - ready to commit"
```

### 2. Auto-Generate Summary
```
Create simple summary:
  Files created:    [count]
  Files modified:   [count]
  Memory updated:   [yes/no]
  Tests affected:   [count]
  
Suggest next step: "Run `/validate-specs` before starting Phase N"
```

### 3. Link Next Actions
```
Based on what was done:
  IF specs changed → Suggest: /spec-change analysis
  IF tests updated → Suggest: npm test run
  IF phase complete → Suggest: /propagate-spec
```

## Configuration

### Allowed Tools (for fast execution)
```
✓ file_search (fast)
✓ grep_search (fast)
✓ read_file (normal)
✓ create_file (normal)
✓ replace_string_in_file (normal)
✓ multi_replace_string_in_file (normal)

✗ run_in_terminal (disabled - manual commits only)
✗ semantic_search (expensive - use grep instead)
```

### Memory Paths
```
User memory:     /memories/
Session memory:  /memories/session/
Repo memory:     .codebase/

Auto-update:
  - SPEC_CHANGELOG.md (after .planning/ changes)
  - MODULE_INDEX.md (after .codex/ changes)
  - CURRENT_STATE.md (daily snapshot)
```

### Token Optimization Rules
```
1. Cache .codebase/CURRENT_STATE.md for 30min
2. Don't re-read files in same command chain
3. Use grep_search instead of semantic_search
4. Reference paths by pattern, not full content
5. Trust cached state unless "changed" keyword used
6. Batch multiple file operations into multi_replace
7. Skip linters, use quick validation only
```

## ⚠️ Token Budget Guards

**Purpose**: Prevent runaway token usage. See `.codex/MODEL_ALLOCATION.md` for model responsibilities.

### Critical Safeguards

#### 🔴 Spec-Impact-Engine (Max 35k)
```
BEFORE auto-propagating spec changes to downstream phases:
  IF tokens_used > 30,000
    → PAUSE execution
    → Prompt: "Propagate changes to [N] phases? (~25k tokens). (YES/NO)"
    → Wait for user response
    → Only proceed if user confirms YES
    
Example: "Propagate API change to Phase 2, 3, 4? (~25k tokens) YES/NO"
```

#### 🔴 Multi-Phase Orchestration (Max 100k)
```
BEFORE recalculating 5+ phases or multi-phase timeline impact:
  IF tokens_used > 80,000
    → PAUSE execution
    → Prompt: "Recalculate 5+ phases? (~80k tokens). (YES/NO)"
    → Wait for user response
    → Only proceed if user confirms YES
```

### Per-Task Budget Limits

| Task | Budget | Check Point | Action |
|------|--------|-------------|--------|
| New feature | 40k | Approaching | Report status |
| Spec impact | 35k | Before propagate | **Ask user** |
| API contract | 20k | Exceeded | Abbreviate output |
| Code review | 18k | Exceeded | Focus high-risk only |
| Design spec | 25k | Exceeded | Report & stop |

### Session Budget Tracking

```
Per session: 200k total

  ✓ 0-50k:   Normal work (GREEN)
  ⚠️ 50-150k: Report checkpoint at 75k
  🔴 150-200k: Confirm before starting major task
  🛑 200k+:  Session limit exceeded
```

**When to report**:
- At 75k tokens: "✓ 37.5% of session used. Next checkpoint: 150k"
- At 150k tokens: "✓ 75% of session used. Suggest: commit work, start fresh session"
- At 190k tokens: "🔴 95% of session used. Large tasks will exceed budget"

### Token Optimization Examples

**Example 1: Spec-Impact Guard in Action**
```
Developer changes API endpoint response format

Codex analysis (~6k tokens used):
  1. Detect change: "Response format changed" (2k)
  2. Find affected: "Phase 2, 3, 4 use this API" (2k)
  3. Estimate cost: "Propagating costs ~25k tokens" (1k)
  4. Total: 6k used, would be 31k total

✓ Codex PAUSES: "Propagate to 3 phases? (~25k tokens) YES/NO"
  
If YES:  Propagate (stays within 35k budget limit)
If NO:   Stop, save 25k for other work
```

**Example 2: Caching Benefit**
```
Working on 3 features in one session

WITHOUT caching:
  Feature 1: Load state (5k) + work (15k) = 20k
  Feature 2: Load state (5k) + work (15k) = 20k
  Feature 3: Load state (5k) + work (15k) = 20k
  Total: 60k ✓

WITH caching (30min TTL):
  Feature 1: Load state (5k) + work (15k) = 20k
  Feature 2: Use cache (0k) + work (15k) = 15k ← Save 5k!
  Feature 3: Use cache (0k) + work (15k) = 15k ← Save 5k!
  Total: 50k ← Save 10k total!
```

```
User: /new-feature "add user roles"

  ✓ Verify contracts/ has entry (skip if new)

```
User implements:

Hook runs:

    - 2026-05-30 14:45: .planning/PHASE_DEPENDENCY_MAP.md - Updated Phase 2 dependencies

## When Hooks DON'T Run

Hooks are **disabled** in these cases:
- Reading files only (no modifications)
- Exploring/searching code (grep, semantic search)
- Simple Q&A (no file changes)
- Pre-planning phase (gathering info)

**Result**: Max speed for research, full safety for changes
